# 工作流操作指南 (VIBE CODE)

> 本文档描述如何通过多个 opencode 会话（每会话一角色）实现自动化文件驱动协作。  
> 所有角色间的通信通过读写对应的 MD 文件完成。

---

## 工作流总览

```
┌──────────┐   发布需求    ┌──────────┐   分配任务    ┌──────────────┐
│ 产品经理   │ ─────────→ │  架构师   │ ──────────→ │ 前端工程师    │
│            │             │          │            │ 后端工程师    │
└──────────┘              └──────────┘            └──────────────┘
      ↑                                                  │
      │                                       代码完成 + 审核通过
      │                                                  │
      │                                       通知测试任务
      │                                                  │
      │                                           ┌──────┴──────┐
      ├────────────────── 测试报告 ────────────── │  测试工程师   │
      │         (写入 §待验收)                    └─────────────┘
      │
  验收通过 → 更新项目进度.md → 迭代完成
```

---

## 操作方法

> 核心原则: **每个 opencode 对话对应一个角色。启动时指定角色文件作为上下文。**

### Step 1: 启动产品经理会话

在新终端中执行：
```
opencode
```
然后输入以下指令：
```
请以产品经理身份工作。首先读取项目根目录的 产品经理.md，了解当前项目状态和需求池。
将下一个迭代的需求写入 产品经理.md 的「当前发布需求」区域，并确保「当前发布需求」区域清晰可被架构师读取。
完成后更新 项目进度.md 的工作流状态。
```
或者直接：
```
请读取 产品经理.md 和 项目进度.md，以产品经理身份发布下一轮需求。
```

---

### Step 2: 启动架构师会话

在新终端中执行（或等待 Step 1 完成后）：
```
opencode
```
然后输入：
```
请以架构师身份工作。首先读取 产品经理.md 的「当前发布需求」区域和 架构师.md。
分析需求对现有架构的影响（记录在 架构师.md 的「需求分析」区域），
将任务拆解并分别写入 前端工程师.md 和 后端工程师.md 的「当前任务」区域。
完成后更新 项目进度.md 的工作流状态。
```

---

### Step 3a: 启动前端工程师会话

```
opencode
```
输入：
```
请以前端工程师身份工作。读取 前端工程师.md 的「当前任务」区域。
实施 UI 开发，完成后更新「代码变更」区域并进行代码自审。
审核通过后更新 测试工程师.md 的「当前测试任务」区域。
```

---

### Step 3b: 启动后端工程师会话

> ⚠️ 可与 Step 3a 并行执行

```
opencode
```
输入：
```
请以后端工程师身份工作。读取 后端工程师.md 的「当前任务」区域。
实施数据层开发，完成后更新「代码变更」区域并进行代码自审。
审核通过后更新 测试工程师.md 的「当前测试任务」区域。
```

---

### Step 4: 启动测试工程师会话

```
opencode
```
输入：
```
请以测试工程师身份工作。读取 测试工程师.md 的「当前测试任务」区域。
根据变更内容编写和执行测试，将测试报告写入「测试报告」区域。
将测试结果反馈到 产品经理.md 的「待验收」区域。
```

---

### Step 5: 回到产品经理验收

```
opencode
```
输入：
```
请以产品经理身份工作。读取 产品经理.md 的「待验收」区域。
根据测试报告决定是否验收通过。通过则更新 项目进度.md 标记迭代完成。
如需优化则发布修正需求。
```

## 快捷指令参考

以下是一行式指令，可直接粘贴到 opencode 中启动各角色：

**产品经理:**
```
读取 产品经理.md 和 项目进度.md。以产品经理身份发布下一轮需求，写入 产品经理.md §当前发布需求，更新 项目进度.md 工作流状态。
```

**架构师:**
```
读取 产品经理.md §当前发布需求 和 架构师.md。以架构师身份分析需求并分配任务到 前端工程师.md 和 后端工程师.md §当前任务，更新 项目进度.md。
```

**前端工程师:**
```
读取 前端工程师.md §当前任务。以前端工程师身份实现代码，完成后更新 前端工程师.md §代码变更，自审后通知 测试工程师.md §当前测试任务。
```

**后端工程师:**
```
读取 后端工程师.md §当前任务。以后端工程师身份实现代码，完成后更新 后端工程师.md §代码变更，自审后通知 测试工程师.md §当前测试任务。
```

**测试工程师:**
```
读取 测试工程师.md §当前测试任务。以测试工程师身份编写和执行测试，写测试报告并反馈到 产品经理.md §待验收。
```

## 工作流规则

1. **单向流转**: 产品经理 → 架构师 → 前后端 → 测试 → 产品经理，不跳过中间角色
2. **文件即合约**: 每个角色的 §当前任务 / §当前发布需求 / §当前测试任务 是交接协议
3. **状态同步**: 每完成一个 Stage，更新 `项目进度.md` 的「工作流状态」表
4. **前后端可并行**: Step 3a 和 Step 3b 可以在两个独立 opencode 会话中同时进行
5. **一处写入，多处读取**: 架构师写入前后端任务后，前后端各自读取各自文件，互不干扰
6. **代码审核为自审**: 当前阶段先通过自审检查清单，后续可引入专门的代码审核角色

## 文件驱动机制说明

每个角色的 MD 文件包含三个关键区域：

```
§输入区域 (📥)    — 由上游角色写入，本角色读取
§工作区域 (⚠️)    — 本角色在此执行工作，更新内容
§输出区域 (📤)    — 本角色写入，下游角色读取
```

具体映射：

| 上游 → 下游 | 上游写入 | 下游读取 |
|-------------|---------|---------|
| 产品经理 → 架构师 | `产品经理.md` §当前发布需求 | `架构师.md` 读取 |
| 架构师 → 前端 | `前端工程师.md` §当前任务 | 同文件 §当前任务 |
| 架构师 → 后端 | `后端工程师.md` §当前任务 | 同文件 §当前任务 |
| 前端 → 测试 | `测试工程师.md` §当前测试任务 | 同文件 §当前测试任务 |
| 后端 → 测试 | `测试工程师.md` §当前测试任务 | 同文件 §当前测试任务 |
| 测试 → 产品经理 | `产品经理.md` §待验收 | 同文件 §待验收 |

## 常见操作场景

### 开始新迭代
以产品经理身份启动 opencode 会话，发布需求。

### 修改需求
在产品经理会话中更新需求，架构师重新分析。

### 修复 Bug
测试工程师发现 Bug → 写入产品经理 §待验收（失败）→ 产品经理发布修复需求 → 正常流转。

### 并行开发
前后端各自独立会话，无需等待对方。测试工程师合并两个角色的交付后统一测试。

### 切换角色
每个角色在独立的 opencode 会话中执行。切换角色即切换终端/会话。也可在同一会话中重新指定角色指令。

### 其他 IDE / 智能体 / CLI
任何支持读取文件上下文的 AI 编码工具，只需:
1. 读取 `项目进度.md` 了解项目
2. 读取对应角色文件并扮演该角色
3. 按工作流协议读写文件完成交接
