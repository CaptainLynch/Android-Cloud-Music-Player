> ⚠️ **架构师在此分析需求对架构的影响**

### F10 本地收藏/喜欢

**现有架构相关模块:**
- `MusicPlayerManager` (object 单例) — 当前持有 ephemeral 播放队列，无持久化
- `SearchScreen` + `SongItem` — 搜索结果列表，无收藏按钮
- `PlayerScreen` — 播放器页面，无收藏按钮
- `NeteaseSettings` (SharedPreferences) — 当前项目无 Room/SQLite

**分析:**
```
需求 F10:
- 涉及模块: UI层（搜索结果列表、播放器页面、新增收藏列表页面） + 数据层（新增 Room 持久化）
- 是否新增模块: 是 — FavoriteEntity (Room 实体)、FavoriteDao、FavoritesRepository、FavoritesScreen (新增页面)
- 是否需要新依赖: 是 — Room (androidx.room:room-runtime + room-ktx + room-compiler KSP)
- 架构风险: 低 — 新增独立模块，与现有搜索/播放逻辑松耦合；Song 包含 List<String> artist 字段，Room 需要 TypeConverter 处理
- 与现有架构的一致性: 一致 — 遵循 MVVM + Repository + StateFlow 模式；须在 MainActivity 中新增导航状态
```

### F12 后台播放支持

**现有架构相关模块:**
- `MusicPlayerManager` (object 单例) — ExoPlayer 生命周期绑定 Activity.onDestroy()，无 MediaSession
- `AndroidManifest.xml` — 无 Service 声明、无 FOREGROUND_SERVICE 权限
- `PlayerScreen` — 播放 UI，StateFlow 驱动
- `build.gradle.kts` — 仅有 `media3-exoplayer` 依赖，需新增 `media3-session`

**分析:**
```
需求 F12:
- 涉及模块: 播放层（MediaSession/MediaSessionService） + 系统层（通知渠道、Foreground Service） + UI层（通知栏 UI）
- 是否新增模块: 是 — MediaPlaybackService (Foreground Service) + MediaSessionManager
- 是否需要新依赖: 是 — media3-session (androidx.media3:media3-session)、可能需要 media3-ui (通知样式)
- 架构风险: 中 — 需要重构 MusicPlayerManager，将 ExoPlayer 所有权从 object 单例迁移到 Service，或使用 MediaSessionService 包装现有播放器；涉及 AndroidManifest 权限声明、通知渠道注册、音频焦点管理
- 与现有架构的一致性: 需调整 — MusicPlayerManager 的 `init(context)` / `release()` 生命周期需适配 Service；当前 `playExternalUrl()` 流程不变，但 UI 层的 StateFlow 订阅需通过 MediaController 桥接
```

### F15 网易云原生加密直连

**现有架构相关模块:**
- `NeteaseApiService` — Retrofit 接口，所有请求使用 `@Query("cookie")` 传递认证信息，目标为 Binaryify 代理服务
- `NeteaseRetrofitClient` — 支持动态 baseUrl 切换，无 CookieJar 配置
- `NeteaseViewModel` — 登录流程调用 `api().loginCellphone(phone, password)`，cookie 持久化到 SharedPreferences 后逐请求拼接
- `NeteaseSettings` — 持久化 cookie / uid / apiUrl
- `NeteaseModels.kt` — 数据模型匹配 Binaryify API 响应格式
- `NeteaseScreen` — 设置页仅有 API URL 输入框，无双模式切换 UI

**分析:**
```
需求 F15:
- 涉及模块: 网络层（新增加密模块、新 Retrofit Service、CookieJar） + UI层（设置页开关）
- 是否新增模块: 是 — WeApiCrypto (AES+RSA+Base64 加密工具)、LinuxApiCrypto (备选加密)、NeteaseOriginApiService (Retrofit 接口，baseUrl=music.163.com)、NeteaseApiStrategy (直连/代理模式切换)
- 是否需要新依赖: 否 — AES/RSA/MD5 加密使用 JDK javax.crypto 标准库（Android SDK 内置），Base64 使用 Android Base64 类
- 架构风险: 高 — 加密算法需与 music.163.com 官方 WEAPI 标准完全一致（AES-128-CBC + RSA 无填充公钥 + MD5 摘要），错误会导致登录失败和所有接口不可用；CookieJar 替代 Query 参数会改变现有请求链路，需保证在代理模式下兼容旧逻辑
- 与现有架构的一致性: 需调整 — NeteaseRetrofitClient 需引入策略模式（ProxyMode vs DirectMode），两种模式使用不同的 Retrofit Service 实例和 OkHttp 拦截器链；NeteaseViewModel 登录/请求流程需根据模式分支
```

### 迭代 12 需求分析

### BUG-005 搜索平台源过滤

**现网问题:**
- GD Studio API `/api.php?types=search&type=kuwo&name=xxx` 传入 `type` 参数后，返回结果的 `source` 字段仍可能被网易云数据污染

**现有架构相关模块:**
- `SearchViewModel.searchMusic()` — 调用 API 后直接赋值 `_searchResults.value = results`，无客户端过滤
- `Song.source` — 已有字段，值为 "netease" / "kugou" / "kuwo" / "migu"
- `MusicApiService.searchMusic()` — 传入 `type` 参数但服务端过滤不可靠

**分析:**
```
需求 BUG-005:
- 涉及模块: 数据层（SearchViewModel 过滤逻辑）
- 是否新增模块: 否
- 是否需要新依赖: 否
- 架构风险: 低 — 客户端侧防御性过滤，不改变 API 调用逻辑
- 与现有架构的一致性: 一致 — 一行 filter 即可
- 修改点: SearchViewModel.searchMusic() 中 `_searchResults.value = results.filter { it.source == platform.value }`
```

**现有架构相关模块:**
- `NeteaseRetrofitClient.getProxyService()` — 支持动态 baseUrl
- `NeteaseSettings.apiUrl` — 用户可配置的 API 地址（SharedPreferences 持久化）
- `NeteaseScreen` 设置页 — 已有 API URL 输入框

**分析:**
```
需求 F16:
- 涉及模块: DevOps 层（Docker/Nginx 部署配置） + 网络层（App 端代理 URL 配置）
- 是否新增模块: 是 — Dockerfile / docker-compose.yml / nginx.conf 部署资产
- 是否需要新依赖: 否 — Docker 和 Nginx 是运行环境，非项目构建依赖
- 架构风险: 低 — 部署操作完全独立于 App 代码；App 端无需代码变更（已有代理模式）
- 与现有架构的一致性: 一致 — App 代理模式无需修改
- 注意事项: 部署域名确定后，App 端设置页可预填新地址作为默认值
```

**分析:**
```
需求 F15降级:

- 是否新增模块: 否 — 当前代码已符合降级要求
- 是否需要新依赖: 否
```

## F10 修正需求分析（测试驳回复验）

> 📥 来源: `产品经理.md` §F10 修正需求（验收阶段驳回）

### Bug 根因分析

开发阶段 `FavoritesRepository` 存在 **数据源双轨分裂**：

| 层级 | 文件 | 问题 |
|------|------|------|
| Room 层 | `AppDatabase.kt` / `FavoriteDao.kt` / `FavoriteEntity.kt` | 已创建但**未接入**业务流程 |
| SharedPreferences | `FavoritesRepository.kt` (object) | **实际使用**，但 SearchViewModel 中错误地按 Room 类 API 调用 |

导致两个数据源 API 不匹配，产生 4 个编译错误：

```
BUG-001: MusicPlayerManager.init()   → 匿名类/apply/函数缺 3 个闭合 `}`
BUG-002: FavoritesRepository(AppDatabase)  → object 单例无法当构造函数调用
BUG-003: .allFavorites / .isFavoriteFlow   → 实际 API 为 .favoriteSongs / .isFavorite()
BUG-004: viewModel.favoriteIds             → SearchViewModel 未暴露此字段
```

### 架构决策: F10 修正

| ID | 日期 | 决策 | 原因 |
|----|------|------|------|
| AD-006 | 2026-05-14 | FavoritesRepository 统一为 Room 单一数据源，移除 SharedPreferences 持久化 | 消除双轨分歧；Room 提供类型安全的结构化查询和 Flow 响应式数据；与 Room 实体/FavoriteDao 已存在的代码资产对齐 |
| AD-007 | 2026-05-14 | FavoritesRepository 保持 `object` 单例模式（非 class 构造注入） | 现有 FavoritesScreen / FavoritesViewModel / MusicPlayerManager 多处直接引用 `FavoritesRepository.xxx`，改为 class 需修改 5+ 处代码；保留 `init(context)` 延迟初始化模式与 `NeteaseSettings` 一致 |
